# Test Go Code

- Focus on unit testing, not integration
- Each `internal/{projectName}/**` has test files
- Use table-driven tests for better readability and maintainability
- Each unit test should mark with t.Parallel(). If any test share state, which is not thread safe, it may still use table driven but with testify suite
- Test file should end with `_test.go`, and package name should be `{packageName}_test`
- Use mockery, the config in `internal/{projectName}/.mockery.yaml`, and run with `pnpm nx gen:mockery {projectName}`
- Those mockery generated often named `mock.go` in the same package. You shouldn't read the whole mock file due to its length. Only read if it is really necessary.
- Mocked interface generated by mockery follow testify style, and in normal package, not test package
- Use testify as testing framework, test suite, and mock
- For those project doesn't follow clean architecture (billing, blog), the test mostly in `internal/{projectName}/core`, else, they would be in `internal/{projectName}/domain` and `internal/{projectName}/app`
- Those project with no clean architecture, the GORM model is also the domain model, which has to use `https://github.com/data-dog/go-sqlmock` for mocking at `database/sql` package

## Run

- Should run `nx lint {projectName}` to lint the syntax of test code, especially best practice from testify
- Use `nx test {projectName}` to run the test
